JC2503 Report

Project Report

This report describes my JC2503 coursework website: a personal introduction site with a multiplayer puzzle game built with Express, EJS, CSS, client-side JavaScript, and Socket.IO.

Overall Website Design

The website has three main pages: the introduction page at the root route, the About page, and the Game page. The first page gives a short personal overview and links to the rest of the site. The About page gives more detail about my study background, technical interests, and project experience. The Game page contains the interactive coursework application. Express handles routing, while EJS templates and shared partials keep the head, navigation bar, and footer consistent.

The visual design adapts a colour theme I often use in personal projects, with yellow, pink, and blue accents across reusable cards, buttons, grids, and panels. Multi-column layouts collapse on smaller screens, and the interface includes a dark/light theme. Accessibility was considered through semantic landmarks, keyboard focus styles, a skip link, alternative text, and ARIA labels.

Game Application Design

The puzzle game is a continuously running turn-based multiplayer application. The server stores the authoritative state in memory, including the 4 by 4 grid, block pool, active player list, current player index, private block, turn timer, scores, and latest event. The block set contains exactly 16 unique blocks, created from four shapes and four colours. At the start of each turn, the server randomly selects one block from the pool and assigns it only to the active player.

Players place their assigned block by clicking an empty board cell. The server then removes the block from the pool and checks horizontal, vertical, and diagonal lines for contiguous runs of three or more blocks sharing shape or colour. Matching blocks are cleared, returned to the pool, and scored at one point per block. This line-clearing check happens before the full-board jackpot check, because a completed line is the immediate result of the placement; the 16 point jackpot is awarded only if no line is cleared and the grid remains full. Players can join at the end of the turn order, leave, disconnect, or be removed after a timeout.

Client and Server Communication

Socket.IO is used because several browsers need live shared state without page reloads. When a browser connects, the server emits a public gameState event containing the board, players, scores, pool count, filled-cell count, timer information, and latest status message. The client sends joinGame with the display name. The server validates it, prevents duplicate active names, adds the player to the turn order, and broadcasts the updated state.

Public state is sent to all players, but the current block is sent through a separate privateTurn event only to the active player's socket. This prevents other players from seeing the block before placement. The client sends placeBlock when a board cell is clicked, and the server checks the player, turn, cell index, and availability. The server also handles leaveGame, disconnect, and timeout removal.

Reflection

The main challenge was keeping game state reliable while users can join, leave, and take turns dynamically. I made the server the only authority for the board, pool, score, and current turn. The browser client does not decide whether a move is legal; it sends user actions and renders server state. This made the program easier to reason about, especially when a player disconnects or times out during their turn.

The interface also changed during testing. At first, the game worked logically, but players did not always have enough feedback about the current turn. I added a shared visible timer, clearer waiting messages, and a short clearing animation so that removed blocks were easier to understand. I also clarified the scoring order in the report: line clearing happens before the full-board jackpot because it is the direct result of the placement. Validating the pages and CSS helped me catch presentation issues early, and it showed that small usability details matter as much as the core game logic.
