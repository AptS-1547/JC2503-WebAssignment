Project Report

This report describes the design and implementation of my JC2503 coursework website: a personal introduction site with an interactive multiplayer puzzle game built with Express, EJS, client-side JavaScript, CSS, and Socket.IO.

Overall Website Design

The website is organised as three main pages: the introduction page at the root route, the About page, and the Game page. The introduction page gives a short personal overview and links to the rest of the site. The About page gives more detailed information about my study background, technical interests, and project experience. The Game page contains the interactive coursework application. I used Express for routing and EJS templates for rendering, with shared partials for the head, navigation bar, and footer.

The visual design follows the frontend style I commonly use in my own personal projects, with yellow, pink, and blue accents as a consistent theme. The CSS defines reusable layouts, cards, buttons, responsive grids, and a dark/light colour mode. The site remains readable on smaller screens by collapsing multi-column layouts into one column. Accessibility was considered through semantic landmarks, keyboard focus styles, a skip link, alternative text, and ARIA labels.

Game Application Design

The puzzle game is a continuously running turn-based multiplayer application. The server stores the authoritative game state in memory, including the 4 by 4 grid, the shared pool of blocks, the active player list, the current player index, the private block, the turn timer, and the latest event. The block set contains exactly 16 unique blocks, created from four shapes and four colours. When a turn begins, the server randomly selects one block from the pool and assigns it only to the active player.

Players place their assigned block by clicking an empty board cell. After placement, the server removes the block from the pool and checks horizontal, vertical, and diagonal lines for contiguous runs of three or more blocks sharing either shape or colour. Matching blocks are cleared, returned to the pool, and scored at one point per removed block. This line-clearing check is applied before the full-board jackpot check, because a completed line is an immediate result of the placement; the 16 point jackpot is therefore awarded only if no line is cleared and the grid remains full. New players join at the end of the turn order, while leaving, disconnecting, or timing out removes a player from the active list.

Client and Server Communication

Socket.IO is used because the game state must be shared between several browser clients without page reloads. When a browser connects, the server emits a public gameState event so the interface can show the board, players, scores, pool count, filled-cell count, and latest status message. The client sends joinGame with the display name entered in the form. The server validates the name, prevents duplicate active names, adds the player to the turn order, and broadcasts the updated state.

The distinction between public and private information is important. Public state is sent to all players through gameState, but the current block is sent with a separate privateTurn event only to the active player's socket. This prevents other players from seeing the block before it is placed. The client sends placeBlock when a board position is clicked, and the server checks the player, cell index, and cell availability. The server also handles leaveGame and disconnect, and uses a turn timer to remove players who do not act in time.

Reflection

The most challenging part was keeping the game state reliable while several users can join, leave, and take turns dynamically. I found it clearer to make the server the only authority for the board, pool, score, and current turn. The browser client therefore does not decide whether a move is legal; it sends user actions and renders the state returned by the server. This reduced client inconsistency.

Another challenge was the scoring logic. The board is small, but checking horizontal, vertical, and diagonal contiguous runs still required a careful representation of possible lines. During multiplayer testing, I noticed that cleared blocks disappeared too quickly, so users might not understand which blocks had been removed in the previous turn. I added a short clearing animation and kept status feedback visible to make each move clearer. The final interface therefore includes a status message, turn order, scoreboard, board counters, timer, and clear feedback. Validation also taught me to distinguish clear HTML or CSS issues from compatibility warnings around CSS custom properties.
